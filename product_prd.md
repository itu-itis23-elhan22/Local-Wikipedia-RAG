## 1. Problem statement

Students need to demonstrate end-to-end understanding of a Retrieval
Augmented Generation (RAG) pipeline. Existing tutorials lean heavily on
hosted LLM APIs (OpenAI, Anthropic). This obscures the moving parts and is
unusable in privacy-sensitive or offline contexts.

We will build a **ChatGPT-style assistant** that answers questions about
famous people and places, using only **local infrastructure**: a local
embedding model, a local vector database, and a local language model. The
user must be able to install dependencies, ingest data, and start chatting
in under 10 minutes on a typical laptop.

---

## 2. Goals & non-goals

### 2.1. Goals

1. **Deterministic, grounded answers.** Every response is derived from
   retrieved Wikipedia chunks; the LLM is forbidden from using outside
   knowledge.
2. **Honest "I don't know."** When relevant chunks are missing, the model
   replies "I don't know" rather than hallucinating.
3. **Domain awareness.** Routing logic identifies whether a query is about
   a person, a place, or both, and applies the matching metadata filter on
   retrieval.
4. **Reproducibility.** A grader can clone the repo, follow the README, and
   reach a working chat UI without additional guidance.
5. **Modularity.** Replacing the LLM, the embedding model, or the vector
   store requires editing a single file (`config.py`) plus a thin wrapper.

### 2.2. Non-goals

- Multi-user authentication or persistence beyond the local laptop.
- Real-time Wikipedia updates (data is snapshotted at ingest time).
- Multi-modal (image / audio) retrieval.
- Production-grade deployment — covered separately in
  `recommendation.md`.

## 4. Functional requirements

### FR-1. Ingest

- The system MUST ingest at minimum the 20 named entities from the
  assignment (10 people + 10 places listed) and total at least 20 people
  + 20 places.
- The system MUST cache raw Wikipedia content locally (SQLite) so repeat
  runs do not hit the network.
- The system MUST be idempotent: rerunning `run_ingest` only fetches what
  is missing, unless `--force` is provided.

### FR-2. Chunk

- The system MUST split each article into chunks suitable for the
  embedding model's context window.
- The chunker MUST support arbitrarily large documents (no fixed array
  bound).
- Chunks MUST overlap by ~15 % to reduce information loss at boundaries.

### FR-3. Embed & store

- Embeddings MUST be generated by a **local** model.
- Storage MUST be a vector database (ChromaDB) persisted to disk.
- The implementation MUST follow **Option B** from the spec: one
  collection, with each chunk tagged `type ∈ {person, place}` in metadata.

### FR-4. Retrieve

- The system MUST classify the user query as person, place, both, or
  unknown using a deterministic rule-based router.
- The retriever MUST apply a metadata filter consistent with the
  classified intent.
- For comparison queries the retriever MUST surface chunks from both sides
  of the comparison.

### FR-5. Generate

- The system MUST use a local LLM via Ollama. The default model is
  `llama3.2:3b`; switching is one env var (`RAG_LLM_MODEL`).
- The system prompt MUST instruct the model to (a) answer only from the
  context and (b) reply "I don't know" when grounding is missing.
- Generation temperature MUST default to ≤ 0.3 to suppress hallucination.

### FR-6. Chat interface

- A web chat UI (Streamlit) MUST allow the user to: ask questions, see
  answers, optionally view the source chunks, and reset the conversation.
- A CLI alternative MUST exist (`python main.py`) for users without
  Streamlit.
- The UI MUST surface the indexed entity count and detected intent for
  every answer.

### FR-7. Documentation

- The repository MUST contain `README.md`, `product_prd.md`,
  `recommendation.md`.
- The README MUST contain installation, ingestion, and runtime
  instructions sufficient for an unfamiliar grader.

## 6. Architecture decisions

### 6.1. One collection, metadata-driven (Option B)

**Why:** A single namespace + boolean metadata filter is more scalable
than parallel collections; we can add new entity types (events,
organizations) without changing the retrieval contract. ChromaDB applies
filters as pre-conditions on HNSW traversal, so the cost is negligible at
this scale.

### 6.2. Sentence-aware overlap chunking

**Why:** Pure fixed-size chunks frequently slice a sentence in half,
which hurts both embedding quality (mid-sentence vectors are noisier) and
LLM grounding. We pick the nearest sentence boundary within a 200-char
window of the target chunk end.

### 6.3. Sentence-Transformers `all-MiniLM-L6-v2`

**Why:** 384-d, 14k sentences/sec on CPU, ~80 MB on disk, strong MTEB
score for retrieval tasks. Avoids running a second Ollama process for
embeddings (`nomic-embed-text` would also be acceptable; switching is one
env var away).

### 6.4. Cosine distance

**Why:** Embeddings are L2-normalized at encode time; cosine is the
mathematically appropriate similarity metric and is more stable than dot
product when working with mixed-length passages.

### 6.5. Rule-based routing

**Why:** Spec explicitly permits keyword/rule-based intent classification.
A second LLM call would double latency without measurable accuracy gain
on a 40-entity corpus. The router maintains an alias index (last-name
fallback for people, salient noun for places) and keyword hints
(*"who"*, *"where"*, *"compare"*).

### 6.6. Strict system prompt + "I don't know"

**Why:** The principal failure mode of small LLMs is confident
hallucination. The system prompt enforces context-only answers and a
canonical refusal phrase, which is also what the spec's failure-case
tests look for.

## 7. Acceptance criteria

A reviewer running the system fresh on a clean checkout MUST observe:

1. `python -m scripts.run_ingest` completes successfully and prints a
   `chunks_added` count > 0.
2. `streamlit run app.py` opens at `localhost:8501`.
3. The query *"What did Marie Curie discover?"* returns a coherent answer
   that mentions polonium and/or radium and cites the *Marie Curie* source.
4. The query *"Who is the president of Mars?"* returns exactly
   `I don't know.` (or a sentence containing that phrase).
5. The sidebar shows the indexed entities and a non-zero vector count.
6. Toggling **Show source chunks** OFF hides the source list without other
   side effects.

---

## 8. Risks & mitigations

| Risk                                              | Mitigation                                                 |
|---------------------------------------------------|------------------------------------------------------------|
| Wikipedia DisambiguationError for rare titles     | Try first candidate; fall back to `auto_suggest=True`      |
| Ollama daemon not running                         | RAGPipeline raises `RuntimeError` with actionable message  |
| Model is too small to follow strict prompt        | Temperature 0.2 + explicit "I don't know" rule + few-shot can be added later |
| Embedding/LLM model drift (HuggingFace deletion)  | Model names are pinned in `config.py`; recommendation.md describes Docker-based pinning |

---

## 9. Out-of-scope (explicit)

- Authentication, ACLs, multi-tenant data isolation.
- GDPR/PII redaction (Wikipedia content is public).
- Mobile apps or browser extensions.
- Streaming Wikipedia diffs.

---

## 10. Future work

See `recommendation.md` for production hardening; in scope for a v1.1:

- Hybrid retrieval (BM25 + vector) for keyword-heavy queries.
- Re-ranker (e.g. `bge-reranker-base`) to lift recall@5 → precision@1.
- Per-user chat history persistence (SQLite).
- Multilingual support (Turkish wikipedia + multilingual MiniLM).
